## DATABASE MODELS

### Channel Model
Represents a Telegram/Bale channel or group connected to a Workspace.

Fields:
- id (UUID, primary key)
- workspace (FK to Workspace)
- added_by (FK to User)
- platform (CharField, choices: telegram, bale, website, linkedin)
- channel_type (CharField, choices: channel, group)
- name (CharField) — display name, e.g. "کانال اخبار ما"
- chat_id (CharField) — Telegram/Bale chat ID, stored after verification
- username (CharField, nullable) — @username if public
- is_verified (BooleanField, default=False)
- verification_code (CharField, nullable) — temporary code used during verification
- verification_expires_at (DateTimeField, nullable)
- is_active (BooleanField, default=True) — soft delete
- created_at, updated_at

### PublishJob Model
Represents a single publishing task (immediate or scheduled).

Fields:
- id (UUID, primary key)
- workspace (FK to Workspace)
- created_by (FK to User)
- content (FK to Content app's Content model, nullable)
- custom_text (TextField, nullable) — if user types directly without using Content model
- media_urls (JSONField, default=[]) — list of image/file URLs to attach
- target_channels (ManyToMany to Channel) — can be multiple channels
- status (CharField, choices: pending, processing, published, failed, cancelled)
- publish_type (CharField, choices: immediate, scheduled)
- scheduled_at (DateTimeField, nullable) — when to publish if scheduled
- published_at (DateTimeField, nullable) — when actually published
- retry_count (IntegerField, default=0)
- max_retries (IntegerField, default=3)
- error_log (TextField, nullable)
- is_active (BooleanField, default=True)
- created_at, updated_at

### PublishResult Model
Stores per-channel result for each PublishJob.

Fields:
- id (UUID, primary key)
- job (FK to PublishJob)
- channel (FK to Channel)
- status (CharField, choices: success, failed)
- platform_message_id (CharField, nullable) — message ID returned by Telegram/Bale
- error_message (TextField, nullable)
- published_at (DateTimeField, nullable)
- created_at

Run migrations after creating models.

## CHANNEL VERIFICATION FLOW

This is how users connect their Telegram or Bale channel/group:

### Step 1 — Request Verification
POST /api/publishing/channels/verify/start/
- Request: { workspace_id, platform (telegram/bale), channel_type (channel/group), name }
- Generate a unique verification_code (format: VERIFY_XXXXXXXX, 8 random uppercase alphanumeric chars)
- Save Channel record with is_verified=False, store verification_code, set verification_expires_at = now + 10 minutes
- Return: { channel_id, verification_code, instructions_text }
- instructions_text (in Persian): "ربات را به کانال/گروه خود اضافه کنید و آن را ادمین کنید، سپس این کد را در کانال/گروه ارسال کنید: VERIFY_XXXXXXXX"

### Step 2 — Bot Polling (Background)
Create a management command: python manage.py run_bot_polling

This command runs a long-polling loop for both Telegram and Bale bots simultaneously using threading:

For each incoming message update:
1. Check if message text matches pattern VERIFY_XXXXXX
2. Query Channel table for matching verification_code where is_verified=False and verification_expires_at > now
3. If found:
   a. Save the chat_id from the update to Channel.chat_id
   b. Try to resolve channel username if available
   c. Set is_verified=True
   d. Clear verification_code
   e. Try to delete the verification message from the channel (bot must be admin)
   f. Send a confirmation message to the channel: "✅ کانال با موفقیت به سامانه متصل شد"
4. If not found or expired: ignore silently

Also add APScheduler job to check every 60 seconds for expired unverified channels and clean them up.

### Step 3 — Check Verification Status
GET /api/publishing/channels/{id}/verify/status/
- Return: { is_verified, verification_code (if still pending), expires_at, status }
- Frontend polls this every 5 seconds while waiting

### Step 4 — Manual re-verify
POST /api/publishing/channels/{id}/verify/retry/
- Reset verification_code and expiry, return new code

## PUBLISHING APIs

### Immediate Publish
POST /api/publishing/publish/now/
- Request:
  {
    workspace_id,
    content_id (optional — from Content model),
    custom_text (optional — if no content_id),
    media_urls (optional list of image URLs),
    channel_ids (list of Channel IDs — can be multiple for simultaneous publish)
  }
- Create PublishJob with publish_type=immediate, status=processing
- Create PublishResult records for each channel
- For each channel, call the appropriate platform service (Telegram or Bale)
- If content_id is provided, get text from Content.body
- Support sending: text only, text+image, text+multiple images, text+file
- Update each PublishResult with success/failed + platform_message_id
- Update PublishJob status: published (if all success), failed (if all failed), or partial
- Return: { job_id, results: [{channel_id, channel_name, status, error}] }

### Scheduled Publish
POST /api/publishing/publish/schedule/
- Same as above but with scheduled_at datetime
- Create PublishJob with publish_type=scheduled, status=pending
- APScheduler will pick it up and execute at scheduled_at time
- Return: { job_id, scheduled_at }

### Cancel Scheduled Job
POST /api/publishing/publish/{job_id}/cancel/
- Only works if status=pending
- Set status=cancelled

### Publish History
GET /api/publishing/history/
- Filter by: workspace_id, channel_id, status, date_from, date_to
- Return paginated list of PublishJobs with their PublishResults
- Include: job_id, content title (if linked), target channels, status, scheduled_at, published_at

GET /api/publishing/history/{job_id}/ — full detail of one job including per-channel results

## PLATFORM SERVICES

Create publishing/services/ directory with:

### telegram_service.py
class TelegramService:
  - Uses TELEGRAM_BOT_TOKEN from environment
  - send_message(chat_id, text) → returns message_id
  - send_photo(chat_id, text, photo_url) → returns message_id  
  - send_media_group(chat_id, text, photo_urls list) → returns message_id
  - send_document(chat_id, text, file_url) → returns message_id
  - delete_message(chat_id, message_id) → bool
  - get_chat(chat_id) → chat info dict
  - All methods use requests library to call Telegram Bot API (api.telegram.org)
  - Handle errors: raise PublishException with Persian message
  - Log all API calls

### bale_service.py
class BaleService:
  - Same interface as TelegramService
  - Uses BALE_BOT_TOKEN from environment
  - Bale Bot API base URL: https://tapi.bale.ai/bot{token}/
  - Bale API is largely compatible with Telegram Bot API
  - Same methods as TelegramService

### publisher.py
class PublisherService:
  - publish(job: PublishJob) — main method, routes to correct platform service
  - Determines platform from Channel.platform
  - Handles retry logic: if fails, retry up to job.max_retries times with 30s delay
  - Updates PublishJob and PublishResult status after each attempt
  - Logs all errors to PublishJob.error_log
